About This Lane

NFT Lane is a derived lane on LaneLayer that processes NFT minting requests. It receives data from Core Lane through the /submit endpoint and processes it according to its internal logic.

This lane follows the LaneLayer pattern: it exposes a single entry point (/submit) that receives all data from Core Lane, processes it, and handles NFT minting operations internally.

API Endpoints

GET /health

Health check endpoint required by LaneLayer CLI. Returns service status and basic information.

POST /submit

The single entry point for receiving data from Core Lane. This endpoint:

  • Accepts raw binary data or JSON
  • Receives metadata via X- prefixed HTTP headers
  • Processes NFT minting requests when JSON contains token_id or metadata fields
  • Handles all data processing for the lane

Example submission format:

{
  "token_id": "1",
  "owner": "0x1234...",
  "metadata": {
    "name": "My NFT",
    "description": "A cool NFT"
  }
}

Headers:

  • X-Forwarded-From - Source of the submission
  • X-User - User identifier
  • X-Content-Type - Content type of the data
  • X-Timestamp - Timestamp of the submission

How It Works

When Core Lane processes a transaction and determines it should be forwarded to this derived lane, it sends the data to the /submit endpoint. The lane then:

  1. Receives the data and metadata headers
  2. Parses the data (JSON or binary)
  3. Processes NFT minting if the data contains NFT-related fields
  4. Stores the NFT in its internal storage
  5. Returns a success response
Note: All data processing happens through the /submit endpoint. There are no separate endpoints for minting or querying NFTs - everything flows through the single entry point as per LaneLayer architecture.

Testing

You can test the /submit endpoint using the LaneLayer CLI:

lane submit --data '{"token_id": "1", "owner": "0x1234...", "metadata": {"name": "Test NFT"}}' \ --header "X-Forwarded-From: core-lane" \ --header "X-User: 0x1234..." \ --header "X-Content-Type: application/json"
